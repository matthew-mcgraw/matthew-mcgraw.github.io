Drone Motion Planning

A from-scratch Python implementation of the SST (Stable Sparse RRT) motion planning algorithm, built as a final project for Advanced Motion Planning at CU Boulder. The goal: plan dynamically-feasible drone trajectories through a dense 3D forestry environment to follow and film a snowboarder on backcountry slopes.

The environment was created in Unreal Engine 5 using Megascans foliage and exported to STL/OBJ via Blender. After 60+ hours wrestling with C++ and OMPL, the entire planner was rebuilt from scratch in Python — integrating ODE-based trajectory propagation, mesh collision detection, and witness-region pruning.

How SST Works

Based on Li, Littlefield & Bekris, IJRR 2014
01

Best_First_Selection

Sample a random state inside the bounding volume. With 5% probability, bias toward the goal. Find the lowest-cost active vertex within δ_BN — if none exist, extend from the closest node instead.

02

MonteCarlo_Prop

Apply random control inputs (ω, α, a) sampled within their bounds, integrated over a random propagation time t ∈ [0, T_prop]. Avoids solving the two-point boundary value problem entirely.

03

State Validity + Collision

Verify every integration step stays inside the bounding mesh and respects pitch/velocity limits. Then check each trajectory point against the environment via sphere proxy collision in trimesh.

04

Witness & Prune

Check if x_new beats its witness region representative. If so, demote x_peer to inactive and prune leaf nodes up the chain — keeping the tree sparse and near-optimal.

Drone Dynamics Model

The drone uses the same 6-DOF nonlinear model from the course mini-project. Trajectories are forward-integrated with scipy.odeint over 10 steps per propagation. Pitch is bounded to ±π/3 and velocity to ±1 m/s to keep the system stable in flight. 

# Drone equations of motion (odeint integration)
def dXdt(X, t):
    x, y, z, yaw, theta, v = X
    return [
        v * cos(yaw) * cos(theta),   # ẋ
        v * sin(yaw) * cos(theta),   # ẏ
        v * sin(theta),              # ż
        omega,                       # ψ̇  (yaw rate control)
        alpha,                       # θ̇  (pitch rate control)
        acc                          # v̇  (linear acceleration control)
    ]
 
# Propagate over random time t ∈ [0, T_prop=8s], 10 integration steps

Environment, Robot & Collision Detection

Unreal Engine 5 Environment

STL / OBJ

LIDAR data for backcountry slopes proved unavailable online, so the environment was built from scratch in Unreal Engine 5 using Megascans tree assets. A flat landscape with dense foliage was exported through Blender as STL and OBJ files. A matching bounding volume mesh was also created in Blender to constrain state sampling.

Drone as a Generalized Sphere

Quadrotor

The drone model (a quadrotor pulled from GrabCad, designed by Logesh Kumar P) is generalized to a sphere using its maximum bounding dimension as the diameter. This accounts for all possible orientations and avoids complex transform math — placing icospheres at trajectory sample points instead of transforming the full mesh.

Collision Checking

Per Trajectory Step

A persistent trimesh.CollisionManager holds the environment mesh throughout the run. At each integration step along a candidate trajectory, a sphere of radius = drone max dimension / 2 is added, tested for internal collision, then removed. State validity (pitch, velocity, bounding volume) is checked first to short-circuit before the costlier collision check.

Results & Parameter Tuning

The non-dynamic baseline planner reached the goal state in 30 seconds to 2 minutes. Adding full drone dynamics drastically increased planning time — the same parameters that took 2 minutes now took 10+ minutes, and the tree barely left the starting area.

Attempt 1

Barely leaves start

Initial dynamic run with δ_s = 0.8, δ_BN = 1, T_prop = 3, N = 2000. The drone was barely able to leave the starting area — far below expectations.

~2–10 min runtime

Attempt 2

Leaves start, not goal

Nearly doubled all parameters: δ_s = 1.5, δ_BN = 2, T_prop = 8, N = 5000. The drone finally left the starting area but didn't get close to the goal. Tightened yaw/pitch bounds to ±π/12 and acceleration to ±0.35 — runtime dropped back to ~2 min.

~2 min runtime

Attempt 3

~3 GB output mesh

N = 50,000 with goal-bias raised to 25%. The output tree was so large (~3 GB) it couldn't be rendered in Python — each sphere had to be exported as a separate STL and assembled in Blender. Still didn't reach the goal, but foliage wasn't blocking completion — more iterations would have gotten there.

>1 hour runtime

Discussion & Future Work

The core challenge is that kinodynamic planning for a drone in a densely obstructed environment is genuinely hard. The random control propagation used by SST generates many trajectories that fail state validity or collision checks before a usable branch is added to the tree — and each failed candidate still costs integration time.

A few directions to explore going forward: starting with sparser foliage to establish a working baseline and scaling up density; replacing Monte Carlo propagation with a more targeted control strategy to avoid solving two-point BVPs; and testing whether a more accurate drone dynamic model would improve tree growth. It's also worth exploring the C++ OMPL implementation of SST to compare performance.

Despite not reaching the final goal, building the SST planner from scratch — including the Best_First_Selection, MonteCarlo_Prop, Is_Node_Locally_the_Best, and Prune_Dominated_Nodes algorithms — and integrating it with a full 3D mesh collision pipeline was a significant achievement. The plan is to continue this work and eventually build it into a more user-friendly Python motion planning library.
